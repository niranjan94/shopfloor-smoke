# Per-task subtasks with completion rollup

Issue: [#108](https://github.com/niranjan94/shopfloor-smoke/issues/108)

## Problem

The tasks list (`app/page.tsx`) treats each task as an atomic unit. There is no
way to break a task into smaller steps, and the status cycle (`todo â†’ in-progress â†’ done â†’ todo`)
is driven entirely by user clicks with no relation to inner progress. We need
nested subtasks per task, persisted in IndexedDB, with a rollup rule that
prevents a parent from being marked `done` while any subtask is still open.

## Goals

- Extend the `Task` data model with a `subtasks` array.
- Backfill `subtasks: []` on all existing rows via an IndexedDB v1 â†’ v2 upgrade.
- Render a nested subtask list inside each task card with add / toggle-done / delete.
- Enforce the completion-rollup rule on the existing `cycleStatus` button: a
  parent task can only enter `done` when every subtask is `done`; otherwise
  advancing from `in-progress` cycles back to `todo` (it cannot reach `done`).
- A task with zero subtasks behaves exactly as today.

## Non-goals

- Subtask reordering, drag-and-drop, or nesting beyond one level.
- Per-subtask metadata (category, priority, due date, description).
- Filter / search / sort over subtasks.
- Server sync, multi-device, or migration tooling beyond the local v1 â†’ v2 step.
- Touching stub pages (`dashboard`, `calendar`, `projects`, `settings`).

## Scope

Single feature, three files: `app/types.ts`, `app/db.ts`, `app/page.tsx`. No
decomposition needed.

## Design

### Data model â€” `app/types.ts`

Add a new interface and one field on `Task`:

```ts
export interface Subtask {
  id: string;
  title: string;
  done: boolean;
}

export interface Task {
  // ...existing fields unchanged...
  subtasks: Subtask[];
}
```

`subtasks` is **required** (not optional). After the v2 migration every row in
IndexedDB has the field, and every new task created through `handleSave`
initializes it to `[]`. Making it required keeps the rollup logic in `page.tsx`
free of `?? []` noise.

A subtask has no `createdAt`/`updatedAt`. Subtasks are owned by their parent
`Task.updatedAt`; mutating the subtask array bumps the parent's `updatedAt`.

### Persistence â€” `app/db.ts`

Bump `DB_VERSION` from `1` to `2`. Extend the existing `onupgradeneeded`
handler to walk the `tasks` store on the v1 â†’ v2 transition and backfill an
empty `subtasks` array on every row that lacks one:

```ts
const DB_VERSION = 2;

request.onupgradeneeded = (event) => {
  const target = event.target as IDBOpenDBRequest;
  const db = target.result;
  const tx = target.transaction!; // versionchange transaction

  if (!db.objectStoreNames.contains(TASKS_STORE)) {
    // ...existing v1 creation logic unchanged...
  }
  if (!db.objectStoreNames.contains(CATEGORIES_STORE)) {
    db.createObjectStore(CATEGORIES_STORE, { keyPath: "id" });
  }

  if (event.oldVersion < 2) {
    const store = tx.objectStore(TASKS_STORE);
    const cursorReq = store.openCursor();
    cursorReq.onsuccess = () => {
      const cursor = cursorReq.result;
      if (!cursor) return;
      const row = cursor.value as Task;
      if (!Array.isArray(row.subtasks)) {
        cursor.update({ ...row, subtasks: [] });
      }
      cursor.continue();
    };
  }
};
```

Notes:

- The cursor walk runs inside the version-change transaction, so the upgrade
  blocks until backfill completes. Callers awaiting `getDB()` see only the
  v2-shaped data.
- The `event.oldVersion < 2` guard makes the migration idempotent: a fresh DB
  created at v2 skips the cursor walk (the store is empty anyway).
- No other `db.*` method changes. `addTask` / `updateTask` already write
  whatever fields the caller hands them, and the keypath/indexes are unchanged.

### UI â€” `app/page.tsx`

#### Initialising new tasks

In `handleSave`, set `subtasks` on the constructed `Task`:

```ts
subtasks: existing?.subtasks ?? [],
```

The form does not collect subtasks at creation time; they are added per-card
after the task exists.

#### `cycleStatus` â€” rollup rule

Change the cycle so the `done` state is gated:

```ts
function allSubtasksDone(task: Task): boolean {
  return task.subtasks.every((s) => s.done);
}

async function cycleStatus(task: Task) {
  const order: Task["status"][] = ["todo", "in-progress", "done"];
  const i = order.indexOf(task.status);
  let nextStatus = order[(i + 1) % order.length];
  // Rollup: cannot enter "done" while any subtask is open.
  if (nextStatus === "done" && !allSubtasksDone(task)) {
    nextStatus = "todo";
  }
  // ...rest unchanged: set completedAt only when nextStatus === "done"...
}
```

A task with zero subtasks: `every` returns `true`, so the cycle behaves
exactly as today.

When the last open subtask is toggled to `done`, the parent's status is **not**
auto-advanced. The rollup is a *constraint*, not an auto-promoter â€” the user
still clicks the parent circle to mark it done. (See Trade-offs.)

When a subtask flips from `done â†’ undone` on a parent that is currently `done`,
the parent is demoted to `in-progress` and `completedAt` is cleared. This is
the inverse of the rollup constraint and keeps the data consistent.

#### Subtask handlers

Add three handlers on `Home` that all funnel through `db.updateTask` and the
`setTasks` reducer:

```ts
async function addSubtask(taskId: string, title: string): Promise<void>;
async function toggleSubtask(taskId: string, subtaskId: string): Promise<void>;
async function deleteSubtask(taskId: string, subtaskId: string): Promise<void>;
```

Contracts:

- **`addSubtask(taskId, title)`** â€” no-op if `title.trim()` is empty. Appends
  `{ id: Date.now().toString(), title: title.trim(), done: false }`. Bumps
  parent `updatedAt`. Persists and updates state.
- **`toggleSubtask(taskId, subtaskId)`** â€” flips `done` on the matching
  subtask. If the flip causes `allSubtasksDone` to become `false` *and* the
  parent is currently `done`, demote the parent to `in-progress` and clear
  `completedAt`. Bumps `updatedAt`. Persists and updates state.
- **`deleteSubtask(taskId, subtaskId)`** â€” removes the subtask. If, after
  removal, `allSubtasksDone` becomes `true` and the parent was `in-progress`,
  the parent's status is **not** changed (consistency with the no-auto-promote
  rule above). Bumps `updatedAt`. Persists and updates state.

All three are pure state transitions over the existing task â€” they read the
current `Task` from local state by `taskId`, compute the next `Task`, and
write it via `db.updateTask` + `setTasks`. No new database methods.

#### Subtask UI inside the task card

A new local component, `SubtaskList`, rendered inside the existing task card
after the metadata badge row:

```ts
function SubtaskList(props: {
  task: Task;
  onAdd: (title: string) => void;
  onToggle: (subtaskId: string) => void;
  onDelete: (subtaskId: string) => void;
}): JSX.Element;
```

Responsibilities:

- Render each subtask as a row with: a small circle toggle (checked when
  `done`), the title (struck through when `done`), and a `Ă—` delete button.
- An inline "Add subtaskâ€¦" text input at the bottom of the list. Pressing
  Enter calls `onAdd(value)` and clears the field. Empty values are ignored.
- Visible regardless of subtask count (so users can add the first one). Renders
  no list when `subtasks.length === 0`, only the input.
- Indented inside the task card body so the nesting reads visually as a
  subtree of the parent. Styling reuses existing utility classes (`input`,
  `btn`, `badge`) â€” no new CSS files.

The parent task card's status circle gets a tooltip suffix when the rollup
blocks `done`: `"Status: In Progress â€” click to advance (subtasks incomplete)"`.
This is the only UX hint that the rollup exists; we do not disable the button.

### Dependency graph for new units

| Unit                         | Depends on                                          |
| ---------------------------- | --------------------------------------------------- |
| `Subtask` (types.ts)         | nothing                                             |
| `Task.subtasks` (types.ts)   | `Subtask`                                           |
| `onupgradeneeded` v2 branch  | `Task` shape from types.ts                          |
| `addSubtask` / `toggleSubtask` / `deleteSubtask` | `db.updateTask`, `tasks` state, `Subtask` shape     |
| `cycleStatus` (modified)     | `allSubtasksDone`                                   |
| `SubtaskList` component      | `Task`, the three subtask handlers                  |

Each unit can be reasoned about in isolation: the migration is pure-IDB, the
handlers are pure state transitions over `Task`, and `SubtaskList` is a
controlled component fed by props.

## Trade-offs

- **Subtask completion does not auto-promote the parent to `done`.** The
  rollup is a one-way constraint (you can't *reach* `done` with open
  subtasks). Auto-promoting would be more "magical" but takes agency away
  from the user â€” for a smoke-test target with a single-user IndexedDB it is
  simpler and less surprising to require an explicit click. The inverse
  demotion (open a subtask â†’ parent leaves `done`) *is* automatic, because
  the alternative is data that violates its own invariant.
- **Subtasks live inline on `Task`, not in a separate object store.** A
  dedicated store would scale better and let us index by parent, but for a
  client-only IndexedDB app with no cross-task subtask queries it would add a
  second transaction to every read/write with no payoff. Rejected.
- **`subtasks` is required, not optional.** Optional would avoid the
  migration walk but would force `?? []` at every read site. The v1 â†’ v2
  cursor backfill runs once per browser and is cheap.
- **No new `db.addSubtask` / `db.updateSubtask` methods.** Subtask mutations
  are modelled as whole-task updates through `db.updateTask`. Adding
  subtask-specific persistence calls would duplicate the round-trip with no
  consistency benefit.

## Testing strategy

This project explicitly has no test suite (per `CLAUDE.md`: *"No tests: this
is a smoke test target, not a tested app. There is no test suite."*). The
only automated quality gates available are the package scripts:

- **Type check** â€” `pnpm exec tsc`. Catches the `Task.subtasks` field becoming
  required, mismatches between the migration's row shape and the `Task` type,
  and prop-type errors on the new `SubtaskList`. This layer is *required* to
  pass.
- **Lint** â€” `pnpm lint`. Catches unused imports / variables introduced by the
  new handlers and component. This layer is *required* to pass.
- **Manual smoke verification via `pnpm dev`** at <http://localhost:3000>:
  1. Fresh browser profile (clean IndexedDB): create a task, add two
     subtasks, toggle one, click the parent's status circle â€” it should reach
     `in-progress` then loop back to `todo` (rollup blocks `done`).
  2. Complete both subtasks, click the parent circle through `in-progress`
     to `done`.
  3. Re-open one subtask: the parent should demote to `in-progress` and lose
     its `completedAt`.
  4. Delete a subtask; reload the page; verify state persisted.
  5. Migration: in DevTools, manually downgrade by deleting the `TodoApp`
     database, re-create it at v1 by checking out `main` briefly and adding
     a task without subtasks, then check out this branch and reload â€”
     `Application â†’ IndexedDB â†’ TodoApp â†’ tasks` should show every row with
     `subtasks: []`.

Unit, integration, and end-to-end test layers are **not applicable** to this
project â€” there is no Jest, Vitest, Playwright, or equivalent harness
installed, and `CLAUDE.md` explicitly designates the project as untested.
Introducing a runner would exceed the scope of this issue.
